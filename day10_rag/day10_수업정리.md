# Day 10 수업 정리: LangChain, LangGraph, RAG

## 1) 오늘 수업의 큰 흐름

이번 수업은 아래 순서로 진행된다.

1. LangChain 기초: LLM 호출 흐름(프롬프트 -> 모델 -> 파서) 이해  
2. Tool/Agent 개념: LLM이 필요할 때 도구를 호출하는 방식 체험  
3. LangGraph 개요: Workflow를 그래프(State 기반)로 구성  
4. RAG 구현: 문서 검색 + 생성 파이프라인 구축  
5. LangSmith: 실행 추적(Tracing)으로 디버깅/관찰

핵심 한 줄: **"LLM 단독 답변"에서 "검색 근거 기반 답변"으로 전환하는 날**

---

## 2) 왜 RAG가 필요한가?

LLM 단독 사용의 대표 한계:

- 최신 정보나 사내 문서 같은 외부 지식을 모름
- 그럴듯하지만 틀린 답(환각)을 할 수 있음
- 도메인 특화 질문에 대한 정확도가 떨어질 수 있음

RAG는 이 문제를 다음 방식으로 해결한다.

- 질문과 관련된 문서를 먼저 검색(Retrieval)
- 검색 결과를 컨텍스트로 넣어 답변 생성(Generation)
- 즉, **기억 기반 답변 -> 근거 기반 답변**

---

## 3) 핵심 개념 정리

- **Embedding(임베딩)**: 텍스트를 벡터로 변환
- **Vector Store**: 임베딩 벡터를 저장하고 유사도 검색
- **Retriever**: 질문과 유사한 문서 조각(top-k) 반환
- **Chunking**: 긴 문서를 적절한 크기로 분할
- **RAG Chain**: 검색 결과 + 프롬프트 + LLM 결합 체인
- **Tool/Agent**: LLM이 필요한 함수(도구)를 선택/호출

---

## 4) LangChain vs LangGraph

### LangChain (직선형 파이프라인에 강점)

- 프롬프트/모델/출력 파서를 체인으로 빠르게 구성
- RAG의 기본 흐름을 구현하기 쉬움
- 단, 복잡한 분기/반복/상태 누적이 커지면 관리가 어려워짐

### LangGraph (그래프 기반 워크플로우)

- Node, Edge, State로 AI Workflow 구성
- 상태(State)를 유지하며 단계별 처리 가능
- retrieve -> generate 같은 RAG 흐름을 구조적으로 표현
- 조건 분기, 반복, 디버깅 관점에서 확장성이 좋음

---

## 5) LangSmith를 왜 쓰는가?

- 체인/그래프 실행 과정을 Trace로 확인 가능
- 실제로 어떤 문서가 검색되어 컨텍스트로 들어갔는지 검증 가능
- "왜 이 답이 나왔는지"를 추적해 디버깅 속도 향상

=> RAG/Agent 개발에서 품질 개선 루프(관찰 -> 수정 -> 재검증)에 필수

---

## 6) 오늘 수업 최종 요약

- LangChain으로 LLM 앱의 기본 흐름을 빠르게 만든다.
- LangGraph로 복잡한 Workflow를 상태 기반으로 안정적으로 확장한다.
- RAG로 외부 지식을 연결해 환각을 줄이고 정확도를 높인다.
- LangSmith로 실행 근거를 확인하며 품질을 개선한다.

결론: **좋은 LLM 앱은 "모델 성능"만이 아니라 "검색 품질 + 워크플로우 설계 + 추적/평가"로 완성된다.**
